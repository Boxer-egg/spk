# spk 音频录制与语音识别测试用例

## 一、音频引擎启动流程测试

### TC-001: Engine 后台线程启动 + 超时保护
**前置条件**: 已选择任意输入设备，未开始录制。
**测试步骤**:
1. 按热键触发录制。
2. 观察 UI 是否卡住超过 3 秒。
**预期结果**:
- 录制应在 3 秒内启动，HUD 显示音量条。
- 若蓝牙设备处于不稳定状态，应抛出 `engineStartTimeout` 错误而非卡死主线程。

### TC-002: 前置硬件格式验证
**前置条件**: 输入设备可用。
**测试步骤**:
1. 触发录制。
2. 查看日志中 `inputNode.inputFormat` 的 sampleRate 和 channelCount。
**预期结果**:
- `sampleRate > 0` 且 `channelCount > 0`。
- 若设备未就绪，应抛出 `noInputAvailable` 错误，不 installTap。

### TC-003: Engine 预热 (prewarm)
**前置条件**: App 刚启动或切换了输入设备。
**测试步骤**:
1. App 启动后等待 2 秒。
2. 首次触发录制，记录启动延迟。
3. 对比未预热时的启动延迟。
**预期结果**:
- 预热后的首次录制延迟应显著低于冷启动（< 200ms）。

## 二、输入设备切换测试

### TC-004: AudioUnitSetProperty 设备绑定
**前置条件**: 有多个输入设备（如内置麦克风和蓝牙耳机）。
**测试步骤**:
1. 在菜单栏选择蓝牙耳机作为输入设备。
2. 触发录制。
3. 查看日志确认设备绑定成功。
**预期结果**:
- 日志应显示通过 `AudioUnitSetProperty` 成功设置设备。
- 不应再出现 `perform(setDevice:)` 反射调用。

### TC-005: 切换设备时重建 Engine
**前置条件**: 已选择一个设备并录制过一次。
**测试步骤**:
1. 录制一次（设备 A）。
2. 切换到设备 B。
3. 再次录制。
4. 查看日志确认 engine 被重建。
**预期结果**:
- 日志中应出现 `rebuildEngine` 或新的 `AVAudioEngine` 实例。
- `inputNode.inputFormat` 应反映设备 B 的格式。

### TC-006: 设备热插拔自动适配
**前置条件**: 正在使用蓝牙耳机录制。
**测试步骤**:
1. 开始录制（蓝牙耳机）。
2. 断开蓝牙耳机。
3. 观察菜单栏设备列表是否自动刷新。
4. 再次触发录制。
**预期结果**:
- 菜单栏设备列表自动刷新。
- Engine 自动预热 (`prewarm`)。
- 录制使用系统默认设备（内置麦克风），不崩溃。

### TC-007: 蓝牙设备路由变化后卡死防护
**前置条件**: 蓝牙耳机连接不稳定。
**测试步骤**:
1. 选择蓝牙耳机。
2. 反复触发录制（模拟蓝牙不稳定）。
3. 观察 App 是否卡死。
**预期结果**:
- 即使蓝牙设备路由变化，`engine.start()` 最多阻塞 3 秒后超时。
- App 不卡死，可继续操作。

## 四、WhisperKit 本地转录测试

### TC-012: WhisperKit 模型下载
**前置条件**: 选择 "WhisperKit (本地)"，模型未下载。
**测试步骤**:
1. 打开设置 > 语音识别，选择 "WhisperKit (本地)"。
2. 选择一个未下载的模型（如 Tiny）。
3. 观察下载进度。
**预期结果**:
- 状态显示 "加载中..."，进度条从 0% 增加到 100%。
- 下载完成后状态变为 "就绪"。

### TC-013: WhisperKit 本地转录 - 中文
**前置条件**: WhisperKit 模型已加载（状态 "就绪"）。
**测试步骤**:
1. 选择 "WhisperKit (本地)"。
2. 语言选择 "简体中文"。
3. 触发录制，说一段中文。
4. 停止录制。
**预期结果**:
- 中文文本正确识别。
- 转录延迟取决于模型大小（Tiny < 1s，Small < 3s）。

### TC-014: WhisperKit 本地转录 - 英文
**前置条件**: WhisperKit 模型已加载。
**测试步骤**:
1. 选择 "WhisperKit (本地)"。
2. 语言选择 "English"。
3. 触发录制，说一段英文。
4. 停止录制。
**预期结果**:
- 英文文本正确识别。
- 使用 `.en` 模型时识别准确度更高。

### TC-015: WhisperKit 模型切换
**前置条件**: Tiny 模型已加载。
**测试步骤**:
1. 在设置中切换到 Small 模型。
2. 观察状态变化。
3. 触发录制验证新模型工作正常。
**预期结果**:
- 状态从 "就绪" 变为 "加载中"，然后回到 "就绪"。
- 新模型的转录结果更准确。

### TC-016: WhisperKit 无网络运行
**前置条件**: 断开网络连接。
**测试步骤**:
1. 确认 WhisperKit 模型已下载。
2. 断开 Wi-Fi。
3. 触发录制。
**预期结果**:
- 转录正常工作，不依赖网络。
- 结果与有网络时一致。

## 五、音频格式与数据处理测试

### TC-017: 多声道设备 downmix
**前置条件**: 使用 aggregate 设备或多声道音频接口。
**测试步骤**:
1. 选择多声道输入设备。
2. 触发录制。
**预期结果**:
- 录制正常，音频被正确 downmix 到单声道。
- WhisperKit 接收 16kHz mono Float32 数据。

### TC-018: 音频格式转换验证
**前置条件**: 使用 48kHz 输入设备。
**测试步骤**:
1. 选择 48kHz 设备。
2. 使用 WhisperKit 录制。
3. 查看日志确认转换后的 sampleRate。
**预期结果**:
- 日志显示目标格式为 16000 Hz。
- 转录结果正确。

## 六、边界与异常测试

### TC-019: 无可用输入设备
**前置条件**: 断开所有麦克风（包括内置麦克风）。
**测试步骤**:
1. 触发录制。
**预期结果**:
- 抛出 `noInputAvailable` 错误。
- App 不崩溃。

### TC-020: 快速连续录制
**前置条件**: 任意 provider 已就绪。
**测试步骤**:
1. 快速连续按热键 5 次（触发-停止-触发-停止...）。
**预期结果**:
- 每次录制正常开始和停止。
- 不泄露 engine 或 provider 实例。

### TC-021: 长时间录制
**前置条件**: 使用 WhisperKit 或 Apple Speech。
**测试步骤**:
1. 持续录制 2 分钟。
2. 停止录制。
**预期结果**:
- 内存不持续增长。
- 转录结果包含完整内容。
